# Changelog 更新策略

本策略用于约束项目级 `changelog.md` 的更新边界，避免噪音记录。

## 仅在以下场景更新

1. 新模块：新增独立模块、子系统、核心目录或可感知的架构单元。
2. 新 Feature：新增用户可感知功能或对外能力。
3. Bug 修复：修复已存在且可复现的问题（功能、逻辑、稳定性、兼容性）。

## 以下场景不更新

1. 纯文案微调（不影响功能语义）。
2. 纯样式微调（不影响交互行为）。
3. 本地工具/个人技能开发（如 `~/.codex/skills/*`）。
4. 过程性实验、临时脚本、草稿内容。
5. 无行为变化的重构或格式化（仅代码整理）。

## 记录格式

按日期分组，推荐使用：

- `Added`：新模块 / 新 Feature
- `Changed`：功能行为调整（非 bug）
- `Fixed`：Bug 修复

每条记录应说明：改动点、影响范围、关键路径（必要时）。

## 执行约束

1. PR/提交包含“新模块 / 新 Feature / Bug 修复”之一时，必须同步更新 `changelog.md`。
2. 不满足上述条件时，不应为了“凑记录”而更新 `changelog.md`。
